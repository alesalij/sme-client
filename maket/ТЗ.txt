– доступна только для сотрудников АМЛ. Поскольку мы платим за вызов метода предлагаю чтобы мы сами вводили ИНН и получали результаты. Иначе могут какие попало инн туда вводить в большом кол-ве.

окно для вывода результатов внутренних проверок контрагентов
окно для вывода консолидированного ответа о результатах проверки (статус «успешно», окно для ввода доп данных каких?, статус «неуспешно»)
окно для вывода результатов внутренних проверок контрагентов по расширенным данным
окно для вывода инфо о сгенерированной анкете
окно для прикрепления скана подписанной анкеты (отдельная вкладка? надо по ИНН найти контрагента и по нему сохранить данные? Можно так)

Логика работы:
Вкладка 1. Проверка по типу запрашиваемых данных 1
1. Пользователь вводит ИНН
2. ИНН отправляется в СПАРК (режим 1)
3. По указанному ИНН из СПАРК возвращается инфо о найденных ИНН
4. Данная инфо выводится в интерфейс
5. Пользователь выбирает корректное ЮЛ, подтверждая дальнейшую работу с ним
6. ID подтвержденного ЮЛ отправляется в СПАРК (режим 2)
7. После получения от БД инфо о сохранении данных о ЮЛ возвращается инфо об успешном или не успешном сохранении расширенных данных
8. После получения от БД инфо о сохранении данных о проверках СПАРК возвращается инфо об успешном или не успешном сохранении данных о проверках.
9. После получения от БД инфо о проведении внутренних проверок вывести инфо об успешном или не успешном сохранении данных о проверках
10. После получения от БД инфо о консолидированном результате проверок вывести инфо об их результате: либо успешно, либо требуются доп данные, либо неуспешно.
11. Пользователь вводит доп данные
12. Доп данные передаем в БД на сохранение
13. После получения от БД инфо о сохранении доп данных возвращается инфо об успешном или не успешном сохранении данных.
14. При получении скана подписанной анкеты передать данные в БД.

Необходимо сохранять инфо о том, какой пользователь какие данные ИНН вводил и какое ЮЛ далее подтвердил.
